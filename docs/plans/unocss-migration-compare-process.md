# UnoCSS 迁移后对比流程

## 目的

这份文档记录本次项目从 WindiCSS/Tailwind 风格原子类迁移到 UnoCSS 后，如何做一轮可复用的对比验证。

核心原则只有两条：

1. 先比原子类，不要一上来就比打包产物体积。
2. 先修语义和生成规则差异，再做页面截图回归。

这次项目里，真正有效的顺序是：

1. 抽取旧原子类结果
2. 生成 UnoCSS 原子类结果
3. 直接对比 CSS 原子类
4. 反查源码里的不兼容写法
5. 做页面截图回归
6. 只保留确认有问题的 diff 图

## 一、原子类对比

### 1. 对比目标

这里要对比的不是 `build` 后 JS/CSS 总包大小，而是：

- 旧方案最终能生成哪些原子类
- UnoCSS 最终能生成哪些原子类
- 哪些类名虽然同名，但语义单位已经变化

本次经验说明，直接对比 CSS 原子类比对比构建产物更有效，因为：

- 打包体积会被代码分割、压缩、treeshaking、业务代码改动干扰
- 原子类对比能更快定位迁移风险点
- 很多问题本质上不是“少生成了类”，而是“同名类单位变了”

### 2. 旧原子类基线

先基于旧配置生成或抽取一份旧原子类基线 CSS，作为迁移前参照。

如果项目还保留 Windi/Tailwind 侧的生成链路，优先产出一份明确的基线文件，例如：

- `tailwind-atomic.css`
- `windi-atomic.css`

如果旧链路已经拆掉，也可以使用迁移前保留的产物文件作为基线。

### 3. 生成 UnoCSS 原子类

本次项目里采用 `@unocss/cli` 直接生成 UnoCSS 原子类文件，而不是通过 `vite build` 间接观察。

推荐命令思路：

```bash
npx @unocss/cli "src/**/*.{vue,js,ts}" --config uno.config.ts -o uno-atomic.css
```

如果项目里模板、脚本、路由或其他目录会携带类名，需要把扫描范围补全，例如：

```bash
npx @unocss/cli "src/**/*.{vue,js,ts}" "index.html" --config uno.config.ts -o uno-atomic.css
```

对比阶段建议固定输出文件名，避免每次生成文件名都变化。

### 4. 直接比较 CSS 原子类

对比重点不是整份文件逐行看，而是看下面三类问题：

- 旧类有，Uno 没生成
- 新旧都生成，但值不一样
- 新旧类名写法相同，但 Uno 的默认语义不是项目想要的

本次迁移里，最典型的问题就是“数字类在旧方案里是 px 语义，但 Uno 默认会按 rem 或 preset 规则解释”。

例如这些类不能只看“有没有生成”，而要看“生成值是不是对的”：

- `text-14`
- `w-400`
- `h-40`
- `mt-10`
- `max-w-80`
- `mb-3`
- `px-5`
- `py-5`
- `py-3`
- `px-30`
- `pr-80`
- `pb-2.5`
- `pl-5`
- `gap-2`
- `gap-2.5`
- `gap-3`
- `gap-4`
- `gap-5`
- `gap-6`

## 二、源码反查规则

### 1. 数字类统一改成显式像素

本次项目里，很多历史写法本质上是“数字即 px”语义，因此迁移时不能继续依赖简写，需要改成显式方括号写法。

典型规则：

- `text-14` -> `text-[14px]`
- `w-400` -> `w-[400px]`
- `h-40` -> `h-[40px]`
- `mt-10` -> `mt-[10px]`
- `ml-22` -> `ml-[22px]`
- `max-w-80` -> `max-w-[80px]`
- `pr-80` -> `pr-[80px]`
- `py-5` -> `py-[5px]`
- `pl-5` -> `pl-[5px]`
- `pb-2.5` -> `pb-[2.5px]`

注意：

- 旧方案里 `mb-12px` 这种写法，在 Uno 里应改成 `mb-[12px]`
- 不要混用 `mb-12px` 这种非 Uno 标准写法

### 2. 同名类不等于同语义

这次迁移里要特别警惕下面这种情况：

```css
.w-400 {
  width: 400px !important;
}
```

旧方案这里是 `400px`，但 Uno 某些 preset 或默认语义下，同名数字类可能会按 spacing scale、rem 或 token 解释。

所以数字类迁移不要偷懒，能显式写成 `-[numberpx]` 的，统一改显式。

### 3. 动态 class 必须单独处理

UnoCSS 不能可靠生成运行时拼出来的动态类。

这类写法必须重点排查：

```vue
:class="form.status === 0 ? '' : 'mb-0'"
```

```vue
:class="`text-[${variables.textColorPrimary}]`"
```

```vue
:class="`w-${width}`"
```

处理原则：

- 条件分支里是静态完整类名的，可以保留，但要确保扫描能命中
- 模板字符串拼接类名的，不要继续依赖旧写法
- 颜色等变量值优先改成 `:style`
- 如果确实要保留原子类表达，改成 `text-[var(--...)]`
- 只在 `uno.config.ts` 里补必须的兼容，不兼容旧动态拼接套路

### 4. 语义类和全局工具类要反查

本次迁移里，还处理了一批全局工具类替换问题，例如：

- `.fr`
- `.fl`
- `.pr-5`
- `.block`
- `.pointer`
- `.text-center`

处理顺序应该是：

1. 先全局搜索这些类的使用点
2. 逐个改成 Uno 原子类
3. 确认仓库里已无使用
4. 再删除全局样式定义

不要反过来先删全局类，否则页面容易出现隐蔽回归。

### 5. 自定义语义类优先直接落回原子类

本次项目里的 `data-access-offset` 就属于这种情况。

如果它最终只是一个固定偏移值，优先直接改为明确的 Uno 类，例如：

- `ml-[120px]`

然后删除旧类名定义，避免保留一层没有必要的语义包装。

## 三、配置收缩原则

### 1. 不要把旧 Tailwind 配置整份搬进 Uno

本次迁移的经验是：

- 能删就删
- 能改回显式类就不要靠 theme token 兜底
- Uno 配置只保留“当前仓库必须依赖”的最小安全集

尤其是这些内容，要优先评估是否能直接删除：

- 从 `tailwind.config.mjs` 整体 `import`
- 旧 `theme.extend` 中只是为了兼容历史数字类的 token
- `important: true`
- 仅为少量历史字符串兜底的大块 `safelist`

### 2. 这次项目里的配置清理结论

以下这类仅为旧数字类兜底的 token，优先通过源码改写清掉，而不是继续保留在 Uno 主题中：

- `spacingTokens`
- `widthTokens`
- `minWidthTokens`
- `heightTokens`
- `fontSizeTokens`

也就是说，像下面这些历史类：

- `text-14`
- `w-400`
- `h-40`
- `mt-10`

应该直接改写成：

- `text-[14px]`
- `w-[400px]`
- `h-[40px]`
- `mt-[10px]`

而不是继续在 `uno.config.ts` 里维护一套大 token 映射。

### 3. 保留最小 safelist

只有在下面场景才建议保留 safelist：

- 第三方组件内部透出的固定类名，源码扫描不到
- 模板中确实存在少量无法静态分析但又稳定的类名

如果只是为了兼容旧写法，优先改源码，不要无限堆 safelist。

## 四、页面回归流程

### 1. 页面回归的作用

原子类对比只能发现“类是否生成、值是否变化”，但不能覆盖：

- 条件渲染缺块
- 结构顺序变化
- 表格内容渲染异常
- 运行时逻辑导致的 UI 缺失

所以页面回归必须做，但应该放在原子类和源码整改之后。

### 2. 页面回归的执行顺序

建议分两层做：

1. 菜单直达页
2. 列表中的二级入口页

二级入口页至少要覆盖：

- `详情`
- `查看`
- `编辑`
- 弹窗型编辑页
- Tab 内部的详情页

### 3. 对比口径

页面比对时，不能把所有差异都当成 UnoCSS 回归。

要区分三类情况：

1. 样式回归
2. 结构/渲染回归
3. 纯数据差异

本次项目里，确认有问题的差异里，很多并不是单纯样式问题，而是：

- 本地缺少整块区域
- 表格渲染成 `[object Object]`
- tab 少了一块内容

这种也要记录，但结论要写清楚是“结构/渲染异常”，不要误写成“样式问题”。

### 4. 只保留有问题的 diff 图

这次项目最终采用的规则是：

- 对比过程中可以生成大量临时截图
- 只有确认有问题的页面，才把 side-by-side 图放到 `docs/ui-compare/diff`
- 其它临时截图和中间拼图全部删除

这样最终归档目录里只保留高价值问题样本。

## 五、推荐执行清单

后续其他项目迁移 UnoCSS 时，建议按下面顺序执行。

### 阶段 1：原子类基线

1. 生成旧方案原子类基线文件
2. 用 Uno CLI 生成 `uno-atomic.css`
3. 对比缺失类、冲突类、同名异义类

### 阶段 2：源码整改

1. 全局搜索数字类
2. 把 px 语义类统一改成 `-[numberpx]`
3. 全局搜索动态 class 拼接
4. 改掉模板字符串类名和运行时拼接类名
5. 搜索全局工具类使用点并替换成 Uno 原子类
6. 删除无用兼容 token、safelist、important 配置

### 阶段 3：页面回归

1. 先验菜单直达页
2. 再补查列表详情/查看/编辑页
3. 对差异做“样式/结构/数据”分类
4. 只归档确认有问题的 diff 图

### 阶段 4：收尾

1. 删除临时截图目录
2. 只保留正式问题图和经验文档
3. 在迁移说明里记录本项目的特殊规则

## 六、本次项目的关键经验

### 1. 不要以打包体积为第一比较对象

迁移初期最有效的是“原子类对比”，不是“构建产物体积对比”。

### 2. 数字类必须明确单位

只要历史项目里数字类是 px 语义，就统一改成方括号显式像素，不要继续赌 preset 的默认解释。

### 3. 动态类是最大风险点

Vue 模板里的动态拼接类名，UnoCSS 不会天然安全兼容，这部分必须专项排查。

### 4. 配置越少越安全

大部分旧 `theme token` 不是必须保留，能通过源码改写消灭的，就不要留在 `uno.config.ts` 里。

### 5. UI 回归不能只看菜单页

很多问题都出现在列表后的二级入口页，尤其是详情、查看、编辑页。

## 七、交付物建议

每次迁移结束后，建议至少保留以下交付物：

- 一份 UnoCSS 迁移对比流程文档
- 一份本项目的迁移注意事项文档
- 一组确认有问题的页面 diff 图

本项目里，页面问题图统一归档在：

- `docs/ui-compare/diff`

## 八、快速执行清单

后续其他项目迁移 UnoCSS 时，可以直接按下面 10 步执行：

1. 先保留一份旧方案原子类基线 CSS，不要先删旧配置。
2. 用 `@unocss/cli` 基于源码生成一份 `uno-atomic.css`。
3. 直接对比旧原子类和 `uno-atomic.css`，先找“缺失类”和“同名异义类”。
4. 全局搜索数字类，把 `text-14`、`w-400`、`mt-10` 这类统一改成 `text-[14px]`、`w-[400px]`、`mt-[10px]`。
5. 全局搜索动态 `:class`、模板字符串类名、运行时拼接类名，改成静态类、`:style` 或 `var(--...)`。
6. 全局搜索 `.fr`、`.fl`、`.pointer`、`.text-center` 这类旧工具类，替换成 Uno 原子类后再删全局定义。
7. 收缩 `uno.config.ts`，只保留当前项目必须的 theme、rule、safelist，不要整份继承旧 tailwind 配置。
8. 先回归菜单直达页，再补查列表里的 `详情 / 查看 / 编辑` 二级入口页。
9. 页面差异要区分“样式回归 / 结构渲染异常 / 纯数据差异”，不要混写。
10. 只把确认有问题的 diff 图归档到 `docs/ui-compare/diff`，其余临时截图全部删除。
