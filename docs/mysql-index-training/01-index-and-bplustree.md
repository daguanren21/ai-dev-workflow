# 第 1 节：索引是什么？B+Tree 原理

## 一、这一节要解决什么问题

这一节重点解决三个问题：

1. 索引到底是什么？
2. 为什么数据库索引常用 B+Tree？
3. 为什么索引能让查询变快？

## 二、索引是什么

可以把索引理解成一本书的目录。

- 没有目录时，你只能一页一页翻。
- 有目录时，你先找到章节，再快速定位页码。

数据库也是一样：

- 没有索引时，通常需要全表扫描。
- 有索引时，可以先在索引结构中定位，再去找数据。

索引的本质是：

> 用额外的空间，换取更少的数据扫描量。

## 三、没有索引时发生什么

假设表里有 100 万行数据，执行下面的 SQL：

```sql
select * from user_profile where name = 'user_01000';
```

如果 `name` 没有索引，数据库可能只能这样做：

```text
第 1 行 -> 第 2 行 -> 第 3 行 -> ... -> 第 1000000 行
```

这就是全表扫描，特点是：

- 扫描行数多
- 查询时间随数据量增长明显变慢
- 很容易拖慢系统

## 四、为什么常用 B+Tree

MySQL InnoDB 常见索引底层结构是 B+Tree。

你不用死抠理论定义，先抓住三个关键点：

1. 树很矮
2. 数据有序
3. 叶子节点可顺序遍历

一个简化示意图如下：

```text
                [M | T]
              /    |    \
        [A-F]   [G-S]   [U-Z]
         /         |        \
   [A,B,C]   [M,N,O,P]   [U,V,W,X]

叶子节点之间还会顺序连接：
[A,B,C] <-> [D,E,F] <-> [G,H,I] <-> ...
```

## 五、B+Tree 为什么适合数据库

### 1. 树高低，查找层数少

由于每个节点可以存很多键值和指针，所以 B+Tree 扇出很大，树通常只有 2 到 4 层。

这意味着查找一次记录，不需要遍历很多层。

### 2. 天然支持范围查询

叶子节点按顺序连接，所以以下 SQL 很适合 B+Tree：

```sql
select * from orders
where create_time >= '2026-03-01'
  and create_time < '2026-03-10';
```

定位到起点后，可以顺着叶子节点往后扫，不需要回头。

### 3. 天然支持排序

因为索引本身有序，所以某些 `order by` 可以直接利用索引顺序，减少额外排序开销。

## 六、为什么不是普通二叉树

普通二叉树在极端情况下会退化成链表，查找效率变差。B+Tree 不一样：

- 每个节点存很多键
- 树更矮
- 磁盘 IO 次数更少

数据库关心的不只是 CPU，而是磁盘页读取成本。B+Tree 非常适合按页读取。

## 七、为什么不是哈希索引

哈希索引适合等值查找，例如：

```sql
where id = 100
```

但它不擅长：

- 范围查询
- 排序
- 最左前缀这种顺序匹配场景

而 B+Tree 同时支持：

- 等值查询
- 范围查询
- 排序
- 前缀匹配

所以数据库里通用性更强。

## 八、索引为什么能快

把一次查询拆开看：

### 没有索引

```text
从头到尾扫描大量行 -> 找到目标记录
```

### 有索引

```text
先在 B+Tree 里定位 -> 快速找到对应数据页 -> 返回结果
```

真正快的原因，不是“索引很高级”，而是：

- 缩小扫描范围
- 降低磁盘 IO
- 减少无效数据访问

## 九、这一节必须记住的话

- 索引不是数据本身，而是帮助查找的数据结构。
- MySQL InnoDB 常见索引底层是 B+Tree。
- B+Tree 适合数据库，是因为它树矮、有序、支持范围查询。
- 索引快的本质是“减少扫描”和“减少 IO”。
